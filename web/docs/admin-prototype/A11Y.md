# A11Y.md — Tulala Admin Accessibility Conventions

> This document defines the keyboard model, ARIA usage, and screen-reader patterns  
> for the Tulala admin prototype. Production must meet WCAG 2.1 AA at minimum.

---

## 1. Keyboard Navigation Model

### Global shortcuts

| Key | Action |
|---|---|
| `Cmd-K` / `Ctrl-K` | Open command palette |
| `Escape` | Close drawer / modal / command palette / popover |
| `Tab` / `Shift-Tab` | Move focus between interactive elements |
| `Enter` / `Space` | Activate focused button or link |
| `↑` / `↓` | Navigate lists (roster rows, inquiry rows) |
| `←` / `→` | Navigate tab bars, sub-nav segments |

### Drawer keyboard model

- **Open trigger** → focus moves to first focusable element inside drawer
- **Escape** → drawer closes, focus returns to trigger
- **Tab cycle** → trapped inside drawer while open (`DrawerShell` manages this)
- **Backdrop click** → closes drawer, focus returns to trigger

### Modal keyboard model

- Same as drawer, but backdrop click should optionally not close (confirm dialogs)
- All modals must have at minimum: title, dismiss button, confirm button

### Roving tabindex (lists)

When a list has too many rows to Tab through efficiently, use `useRovingTabindex()`:

```tsx
import { useRovingTabindex } from "./_primitives";
const { rowProps, onKeyDown } = useRovingTabindex(items.length);
// rowProps(i) → { tabIndex, onKeyDown } for each item
```

- Only one row in the list has `tabIndex=0` at a time
- Arrow keys move focus between rows without leaving the list
- `Enter` activates the focused row

## 2. ARIA Conventions

### Page-level

```tsx
// Main content area
<main role="main" aria-label="Roster">

// Navigation rail
<nav aria-label="Main navigation">

// Sidebar panel
<aside aria-label="Inquiry rail">
```

### Interactive elements

**Always use semantic HTML for interactive elements:**

```tsx
// ✅ Button for actions
<button type="button" onClick={...}>Send offer</button>

// ✅ Link for navigation
<a href="/talent/marta-reyes">View profile</a>

// ❌ Wrong — div with onClick
<div onClick={...}>Send offer</div>
```

**When a custom interactive element is unavoidable:**

```tsx
<div
  role="button"
  tabIndex={0}
  aria-label="Open inquiry RI-202"
  onKeyDown={(e) => (e.key === "Enter" || e.key === " ") && onClick()}
  onClick={onClick}
>
```

### Disclosure / accordion

```tsx
<button
  aria-expanded={isOpen}
  aria-controls="panel-id"
  onClick={() => setIsOpen(!isOpen)}
>
  Section title
</button>
<div id="panel-id" hidden={!isOpen}>
  Panel content
</div>
```

### Live regions (toasts, status updates)

```tsx
// Toast container — polite for non-critical, assertive for errors
<div role="status" aria-live="polite" aria-atomic="true">
  {currentToast}
</div>

// Error messages
<div role="alert" aria-live="assertive">
  {errorMessage}
</div>
```

### Activity feeds / message streams

```tsx
// Message log — additions are announced; deletions are not
<div role="log" aria-live="polite" aria-label="Conversation thread">
  {messages.map(...)}
</div>
```

### Loading states

```tsx
// Skeleton / loading area
<div aria-busy="true" aria-label="Loading roster...">
  <PageSkeleton />
</div>
```

### Modals

```tsx
<div
  role="dialog"
  aria-modal="true"
  aria-labelledby="modal-title"
  aria-describedby="modal-desc"
>
  <h2 id="modal-title">Archive talent</h2>
  <p id="modal-desc">This will hide them from all public rosters.</p>
</div>
```

## 3. Focus Rings

**Never remove the focus ring.** Use the brand focus style:

```tsx
// In global CSS (not inline styles):
:focus-visible {
  outline: 2px solid var(--color-brand, #0F4F3E);
  outline-offset: 2px;
}

// In inline-style components (prototype compromise):
":focus-visible": {
  outline: `2px solid ${COLORS.accent}`,
  outlineOffset: 2,
}
```

The prototype uses `outline: "none"` on some custom interactive elements — this is a known P1 accessibility debt (see ROADMAP §3.12.x).

## 4. Color Contrast Requirements

| Context | Minimum ratio |
|---|---|
| Body text on white | 4.5:1 (AA) |
| Large text (18px+ or 14px bold) | 3:1 (AA) |
| UI components / icons | 3:1 (AA) |
| Status chip text on soft fill | Check each combination |

**Known current risks:**

| Token pair | Approx. ratio | Risk |
|---|---|---|
| `inkDim` on `card` | ~3.0:1 | Marginal for timestamps — use sparingly |
| `amber` on `amberSoft` | ~3.5:1 | OK for large text, check at 11px |
| `indigo` on `indigoSoft` | ~3.8:1 | OK for labels |

## 5. Screen Reader Patterns

### Icon-only buttons

Every icon-only button must have a text label — either `aria-label` or a visually-hidden `<span>`:

```tsx
<button type="button" aria-label="Close drawer">
  <Icon name="x" size={16} />
</button>
```

### Decorative icons

Icons that are purely decorative (next to text that already conveys the meaning) should be hidden:

```tsx
<Icon name="mail" size={14} aria-hidden />
<span>Inbox</span>
```

### Status chips and badges

Chips communicate state visually. Screen readers need the text content to convey meaning:

```tsx
// ✅ Text in chip is readable
<span>Offer sent</span>

// ❌ If the chip is icon-only, add aria-label
<span role="status" aria-label="Offer sent">✓</span>
```

### Initials avatars

```tsx
<Avatar initials="MR" aria-label="Marta Reyes" />
```

## 6. Form Accessibility

```tsx
// Every input must have a label
<label htmlFor="talent-name">Talent name</label>
<input id="talent-name" type="text" />

// Error state
<input
  id="talent-name"
  aria-invalid={hasError}
  aria-describedby={hasError ? "name-error" : undefined}
/>
{hasError && <span id="name-error" role="alert">Name is required</span>}

// Required fields
<input aria-required="true" />
```

## 7. Keyboard Traps (intentional)

The following surfaces intentionally trap keyboard focus:

| Surface | Trap scope | Escape behavior |
|---|---|---|
| Drawer | Drawer panel | Esc → close, return focus to trigger |
| Modal (confirm) | Modal panel | Esc → cancel (treat as "No") |
| Command palette | Palette panel | Esc → close, return to prior focus |

No other surfaces should trap focus.
